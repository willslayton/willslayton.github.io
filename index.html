Probably my favorite class this semester has been a course in Microprocessor Systems which deals with coding for ARM microcontrollers like the STM32L476 Discovery, which is used in the class, on a low level. Using the MDK ARM software Keil uVision 5 I've been learning about how microprocessors function and have been writing code in C and Assembly to work with the board. A wonderful example of this sort of work is the CMSIS definition of HAL GPIO functions that replaced them in

stm32l4xx_hal_gpio.c

as shown below.


GPIO_PinState CMS_GPIO_ReadPin(GPIO_TypeDef* GPIOx, uint16_t GPIO_Pin){
 if(GPIOx->IDR & GPIO_Pin){

// If the pin is currently HIGH on IDR


  return GPIO_PIN_SET;

// Return HIGH


 } else {
  return GPIO_PIN_RESET;

// Return LOW


 }
}
GPIO_PinState CMS_GPIO_WritePin(GPIO_TypeDef* GPIOx, uint16_t GPIO_Pin, GPIO_PinState PinState){
 if(PinState == GPIO_PIN_SET){

// If the pin is HIGH


  GPIOx->ODR |= GPIO_Pin;

// OR the ODR and pin position


 } else {
  GPIOx->ODR &= !GPIO_Pin;

// AND the ODR and NOT pin position


 }
}
GPIO_PinState CMS_GPIO_TogglePin(GPIO_TypeDef* GPIOx, uint16_t GPIO_Pin){
 GPIOx->ODR ^= !GPIO_Pin;

XOR ODR with NOT pin position


}

This code defines functions that can read the value of a single pin, write to the value of a single pin, and toggle the value of a single pin using only the base GPIO function, compared to premade functions with HAL, as well as some clever boolean algebra. Although this is dealing in a quite low level space with signals from single GPIO pins on a microcontroller, but the class goes even lower than that. Assembly is one of the lowest level languages that allows for improvements over even languages like C in processing time, which is important when thinking in terms of microcontrollers. Below is a simple implementation of a function that runs a for loop that calculates the sum of numbers 0 to 9, a task which would only take a few lines in C.


func_for_loop PROC
 MOV r0, #0

; i = 0


 MOV r1, #0

; total_sum = 0


loop CMP r0, #10

; comparing i to 10


 BGE endloop

; if i >= 10 branch endloop


 ADD r1, r1, r0

; total_sum += i


 ADD r0, r0, #1

; i++


 B loop 

; branch loop


endloop
 BX lr
ENDP

Although it's much longer this is the same code that a C assembler may convert it into, which allows you to employ some useful tricks which can save lines of Assembly and in turn processing time. I find things like these fascinating to learn about and fiddle with because this is in all practicality the closest to machine code most people will ever get, and you can see how higher level languages are able to create more complex features from these simple base structures. Another example of these Assembly functions in action is one that involves loading data from memory to a register and storing data from register to memory.


task PROC
 PUSH {r4-r5, lr}

; r0 is the address of an array with halfword sized data
 ; r1 is the address of an array with word sized data
 ; r2 is a uint32_t word


 SUB r2, r2, #1

; counter--


 MOV r3, #0

; moving 0 into r3, think of this as i


task_loop
 CMP r3, r2

; compare r3 to r2


 BGE task_end

; if r3 >= r2 then branch task_end


 LDRH r4, [r0], #2

; load into r4 the value at first array pointer, then increment the array pointer


 LDRH r5, [r0]

; load into r5 the value at first array pointer which is in the new position


 ADD r4, r5, LSL #4

; add to r4, r5 bit shifted left 4 digits (or multiplied by 16)


 STR r4, [r1, r3, LSL #2]

; store data in r4 to the memory address of the second array at index r3


 ADD r3, #1

; increment r3, or i++


 B task_loop

; branch task_loop


task_end
 POP {r4-r5, pc}
 ENDP


The comments should hopefully explain the code well enough, but if it's unclear this is another sort of example of a for loop where the loading and storing of data takes place. One of the most important things to keep in mind when writing functions like this that you find out quickly is to keep track of what size and type data you're dealing with in the registers because there's different native ways to handle them in ARM. There's different instructions such as

LDRB

for loading a byte,

LDRSH

for loading signed halfwords, and

LDRD

for loading doublewords just to name a few. Overall this course has really taught me about how processors work and understand the code that we write and has made me think about how assembly of some languages I code in regularly works.

During my time in the Academy of Information Technology and Robotics (AITR) there were many projects woven into the curriculum involving all sorts of STEM topics like cybersecurity, robotics, and programming. Most of these "challenges" lasted for six weeks, but some were so involved that twelve weeks was required. Below is a list of some of the most interesting challenges that my peers and I worked on back in high school.

  • Cybersecurity CTF


    To teach our class about cybersecurity the academy teachers arranged a six week challenge which culminated in a CTF (capture the flag) competition between student teams. In a CTF each team is given an image on a virtual machine that has several security flaws or vulnerabilities (known as flags) that the team is tasked with finding and fixing. AITR has close ties to our school's CyberPatriot team, which competes in state and national CTFs, so we were visited by guest speakers from the team to teach us the basics of computer security.
